# GBC Poker — Texas Hold'em для Game Boy Color

План реализации для Claude Code. Проект: полноценный Texas Hold'em против 3 ИИ-оппонентов, компилируется в `.gbc` ROM через GBDK-2020, запускается на эмуляторах и на R36S (ArkOS, папка `roms/gbc`).

---

## 1. Технический контекст и ограничения железа

Целевая платформа — Game Boy Color:

- CPU: Sharp SM83 (Z80-подобный), 8 МГц. **Нет аппаратного умножения и деления.** Избегать `*`, `/`, `%` в горячих циклах; деление на степени двойки — сдвигами.
- RAM: 32 KB WRAM (GBC). Стек маленький — не делать глубокой рекурсии и больших локальных массивов.
- ROM: банки по 16 KB. Начать с 64 KB (4 банка), этого хватит с запасом.
- Типы: использовать `uint8_t` / `int8_t` везде, где возможно. 16-битная арифметика заметно дороже 8-битной. `float` запрещён полностью.
- Графика: тайлы 8×8, фон 32×32 тайла, до 40 спрайтов (10 на строку). GBC: 8 палитр фона + 8 палитр спрайтов, по 4 цвета.
- Сохранения: внешняя SRAM (MBC5+RAM+Battery в заголовке ROM), эмуляторы создадут `.sav`.
- Ввод: D-pad, A, B, Start, Select через `joypad()` из gbdk.

Инструменты:

- **GBDK-2020** (последний релиз с GitHub) — компилятор SDCC + библиотеки `gb/gb.h`, `gb/cgb.h`.
- Эмулятор для разработки и отладки: **Emulicious** (лучший дебаггер) или SameBoy. Тест в CI — headless-прогон логики можно собрать нативно (см. §6).
- Графика: тайлы конвертировать через `png2asset` (входит в GBDK-2020).

## 3. Представление данных

Карта — один байт: `card = suit * 16 + rank`, т.е. верхний ниббл масть (0–3), нижний ранг (2–14, туз = 14). Это позволяет извлекать ранг и масть маской/сдвигом без деления:

```c
#define RANK(c) ((c) & 0x0F)
#define SUIT(c) ((c) >> 4)
```

Ключевые структуры:

```c
typedef struct {
    uint8_t hole[2];        // карманные карты
    uint16_t stack;         // фишки (макс 65535 достаточно)
    uint16_t bet;           // ставка на текущей улице
    uint8_t flags;          // FOLDED | ALLIN | ...
    uint8_t persona;        // индекс профиля ИИ (0 = человек)
} Player;

typedef struct {
    Player players[4];
    uint8_t board[5];       // общие карты
    uint8_t board_count;    // 0/3/4/5
    uint16_t pot;
    uint16_t cur_bet;       // текущая ставка для колла
    uint8_t dealer;         // позиция баттона
    uint8_t street;         // PREFLOP/FLOP/TURN/RIVER
    uint8_t sb, bb_level;   // блайнды (индекс в таблице уровней)
} GameState;
```

## 4. Ядро: оценка комбинаций (первый этап)

Задача: из 7 карт (2 карманные + 5 борда) получить сравнимое 16-битное значение силы лучшей 5-карточной руки. Больше — сильнее.

Подход без lookup-таблиц на весь спектр (они не влезают в комфортный бюджет ROM в наивном виде) — **гистограммный оценщик**: считаем количество карт каждого ранга и каждой масти, из этого выводим категорию и кикеры. Это O(n) по картам, без комбинаторики C(7,5), и отлично работает на 8-битном CPU.

Формат результата: `(category << 12) | tiebreak`, где категория 0–8 (high card … straight flush), а tiebreak упаковывает решающие ранги.

Референсная реализация (положить в `src/eval.c`, адаптировать при необходимости):

```c
#include <stdint.h>
#include "cards.h"

/* Категории */
#define HAND_HIGH       0
#define HAND_PAIR       1
#define HAND_TWO_PAIR   2
#define HAND_TRIPS      3
#define HAND_STRAIGHT   4
#define HAND_FLUSH      5
#define HAND_FULL_HOUSE 6
#define HAND_QUADS      7
#define HAND_STR_FLUSH  8

/* Возвращает старший ранг стрита по битовой маске рангов,
   0 если стрита нет. Маска: бит r установлен, если ранг r присутствует (2..14).
   Колесо A-5: туз дублируем в бит 1. */
static uint8_t straight_high(uint16_t mask) {
    uint8_t r;
    if (mask & (1u << 14)) mask |= (1u << 1);   /* A как единица */
    /* ищем 5 подряд, сверху вниз */
    for (r = 14; r >= 5; r--) {
        uint16_t need = 0x1Fu << (r - 4);
        if ((mask & need) == need) return r;
    }
    return 0;
}

/* 7 карт -> сила руки. cards[7], формат card = suit*16 + rank */
uint16_t eval7(const uint8_t *cards) {
    uint8_t rank_cnt[15];   /* [2..14] */
    uint8_t suit_cnt[4];
    uint16_t rank_mask = 0;
    uint8_t i, r, s;

    for (i = 0; i < 15; i++) rank_cnt[i] = 0;
    for (i = 0; i < 4; i++)  suit_cnt[i] = 0;

    for (i = 0; i < 7; i++) {
        r = RANK(cards[i]);
        s = SUIT(cards[i]);
        rank_cnt[r]++;
        suit_cnt[s]++;
        rank_mask |= (uint16_t)1 << r;
    }

    /* --- флеш / стрит-флеш --- */
    for (s = 0; s < 4; s++) {
        if (suit_cnt[s] >= 5) {
            uint16_t fmask = 0;
            uint8_t top5 = 0, cnt = 0;
            uint16_t kick = 0;
            for (i = 0; i < 7; i++)
                if (SUIT(cards[i]) == s)
                    fmask |= (uint16_t)1 << RANK(cards[i]);
            r = straight_high(fmask);
            if (r) return ((uint16_t)HAND_STR_FLUSH << 12) | r;
            /* обычный флеш: 5 старших рангов масти, по 3 бита на кикер не влезает,
               берём топ-ранги в 12 бит (по ниблу мало, упакуем 5x ранги суммой позиций) */
            for (r = 14; r >= 2 && cnt < 5; r--)
                if (fmask & ((uint16_t)1 << r)) { kick = kick * 15 + r; cnt++; }
            /* kick < 15^5? нет, но нам хватает сравнения: используем 12 бит топ-3 */
            (void)top5;
            kick = 0; cnt = 0;
            for (r = 14; r >= 2 && cnt < 3; r--)
                if (fmask & ((uint16_t)1 << r)) { kick = (kick << 4) | r; cnt++; }
            return ((uint16_t)HAND_FLUSH << 12) | kick;
        }
    }

    /* --- каре / фулл-хаус / сеты / пары --- */
    {
        uint8_t quads = 0, trips = 0, pair_hi = 0, pair_lo = 0;
        for (r = 14; r >= 2; r--) {
            if (rank_cnt[r] == 4 && !quads) quads = r;
            else if (rank_cnt[r] == 3 && !trips) trips = r;
            else if (rank_cnt[r] >= 2) {
                if (!pair_hi) pair_hi = r;
                else if (!pair_lo) pair_lo = r;
            }
        }
        if (quads) {
            for (r = 14; r >= 2; r--)
                if (r != quads && rank_cnt[r]) break;
            return ((uint16_t)HAND_QUADS << 12) | ((uint16_t)quads << 4) | r;
        }
        if (trips && (pair_hi))
            return ((uint16_t)HAND_FULL_HOUSE << 12) | ((uint16_t)trips << 4) | pair_hi;

        r = straight_high(rank_mask);
        if (r) return ((uint16_t)HAND_STRAIGHT << 12) | r;

        if (trips) {
            uint16_t kick = 0; uint8_t cnt = 0;
            for (r = 14; r >= 2 && cnt < 2; r--)
                if (r != trips && rank_cnt[r]) { kick = (kick << 4) | r; cnt++; }
            return ((uint16_t)HAND_TRIPS << 12) | ((uint16_t)trips << 8) | kick;
        }
        if (pair_hi && pair_lo) {
            for (r = 14; r >= 2; r--)
                if (r != pair_hi && r != pair_lo && rank_cnt[r]) break;
            return ((uint16_t)HAND_TWO_PAIR << 12)
                 | ((uint16_t)pair_hi << 8) | ((uint16_t)pair_lo << 4) | r;
        }
        if (pair_hi) {
            uint16_t kick = 0; uint8_t cnt = 0;
            for (r = 14; r >= 2 && cnt < 2; r--)   /* 2 кикера, 3-й не влезает в 16 бит */
                if (r != pair_hi && rank_cnt[r]) { kick = (kick << 4) | r; cnt++; }
            return ((uint16_t)HAND_PAIR << 12) | ((uint16_t)pair_hi << 8) | kick;
        }
        {
            uint16_t kick = 0; uint8_t cnt = 0;
            for (r = 14; r >= 2 && cnt < 3; r--)
                if (rank_cnt[r]) { kick = (kick << 4) | r; cnt++; }
            return ((uint16_t)HAND_HIGH << 12) | kick;
        }
    }
}
```

**Важно для Claude Code:** в упаковке кикеров выше сознательный компромисс — в 16 бит влезают не все кикеры (у пары их формально 3, у старшей карты 5). Для абсолютной корректности сравнения на шоудауне добавить функцию `eval7_full()` c 32-битным результатом (использовать `uint32_t` только на шоудауне, не в ИИ-циклах) либо тай-брейк функцией второго уровня. Написать нативные тесты (см. §6), которые ловят такие коллизии, и починить упаковку до перехода к следующим этапам. Убрать мёртвый код с `top5` — он остался как иллюстрация проблемы.

## 5. Модули и этапы (порядок работы для Claude Code)

### Этап 0 — окружение
1. Установить GBDK-2020, собрать hello-world ROM, проверить запуск в Emulicious.
2. Makefile: цели `make rom` (сборка .gbc), `make test` (нативные тесты gcc), `make run` (эмулятор).

### Этап 1 — ядро логики (без графики, тестируется нативно)
1. `cards.h`, `deck.c` — колода 52, тасование Фишера-Йетса на xorshift-PRNG.
2. `rand.c` — 16-бит xorshift; сид собирать из `DIV_REG` в момент нажатий кнопок в меню (на нативных тестах — из аргумента).
3. `eval.c` — оценщик из §4, доведённый до корректности.
4. `game.c` — стейт-машина раздачи: блайнды → префлоп → флоп → тёрн → ривер → шоудаун; корректный расчёт банка, **side-pots при олл-инах** (обязательно), выплаты при сплитах.
5. Полное покрытие тестами: известные руки, сравнение рук, распределение side-pots.

### Этап 2 — ИИ
Профили персонажей: таблица параметров `{ aggression, looseness, bluff_freq, tilt }` в 8-битных значениях 0–255.

Решение на улице:
1. Сила руки: префлоп — таблица Чена или упрощённая матрица 13×13 в ROM (169 байт); постфлоп — `eval7()` от текущих известных карт + грубая оценка потенциала (флеш-дро/стрит-дро по маскам, без симуляций Монте-Карло — дорого для CPU).
2. Скор = сила ± looseness-порог; действие выбирается порогами: fold / call / raise, размер рейза из таблицы (доли банка, степени двойки).
3. Блеф: с вероятностью `bluff_freq` (сравнение с `rand8()`) поднимать со слабой рукой на поздних улицах при малом числе оппонентов.
4. Немного памяти: если человек часто фолдит на рейзы — повышать агрессию (один счётчик на игрока).

Три персонажа: тайтово-пассивный «Профессор», лузово-агрессивный «Ковбой», сбалансированная «Акула».

### Этап 3 — UI
1. Тайлсет карт: ранги и масти как отдельные 8×8 тайлы, карта = метатайл 2×3. Рубашка, подсветка выигрышной комбинации на шоудауне.
2. Экран стола: 4 места, банк, ставки, индикатор баттона/хода. Фон — тайлкарта, карты игроков — спрайты или окно.
3. Меню действий: Fold / Check-Call / Raise (выбор размера D-pad'ом), контекстные подписи.
4. Портреты оппонентов 32×32 + короткие реплики (текстовое окно) на события: рейз, блеф вскрыт, проигрыш.
5. Анимации минимальные: раздача карт (сдвиг спрайта), фишки в банк. Ничего дорогого.
6. GBC-палитры: красные/чёрные масти, зелёное сукно. Проверить читаемость и на DMG-совместимость можно не тратиться — целимся только в GBC (`LCDC`/заголовок ROM: CGB-only допустимо).

### Этап 4 — режимы и мета
1. **Турнир**: 4 игрока, растущие блайнды по таблице уровней (каждые N раздач), игра до вылета всех ИИ; выбывание — экран с местом.
2. **Кэш-режим**: банкролл сохраняется в SRAM, ребай из банкролла.
3. Статистика в SRAM: раздач сыграно, выиграно, лучшая рука, чемпионства.
4. Настройки: скорость анимаций, звук вкл/выкл.

### Этап 5 — звук и полировка
1. Звуки на 4 каналах GB APU: раздача, фишки, победа (короткие джинглы). Можно hUGETracker + hUGEDriver.
2. Титульный экран, экран победы в турнире.
3. Прогон на реальном железе/R36S: проверить производительность на шоудауне (худший случай — оценка 4 рук + side-pots в один кадр; при необходимости размазать по кадрам).

## 6. Тестирование

Ключевой приём: вся логика (deck/eval/game/ai) пишется в **чистом C без зависимостей от gbdk**, чтобы компилироваться и нативно. `tests/test_eval.c` собирается обычным gcc и гоняет:

- табличные кейсы всех 9 категорий, включая колесо A-2-3-4-5 и стрит-флеш;
- сравнение рук с одинаковой категорией (кикеры!) — именно здесь всплывёт проблема упаковки из §4;
- рандомизированный кросс-чек: миллион случайных пар 7-карточных рук, сравнение `eval7` с эталонным медленным оценщиком (полный перебор C(7,5) с честным сравнением) — эталон тоже написать в тестах;
- side-pots: сценарии с 2–3 олл-инами разного размера.

В ROM-сборке эти файлы компилируются SDCC без изменений (следить за отсутствием `malloc`, VLA, рекурсии, float).

## 7. Определение готовности (Definition of Done)

- `make test` зелёный, кросс-чек оценщика без расхождений.
- ROM собирается < 128 KB, стабильные 60 FPS вне переходов.
- Полный турнир от титульника до победы проходится без софтлоков.
- Сейв переживает выключение (проверка в Emulicious и на R36S).
- Раздача с тремя олл-инами делит банк корректно (ручной тест-сценарий с фиксированным сидом).
